Spec-Driven Development para equipos que usan IA

Una metodología y CLI que convierte tu asistente de IA en un proceso de ingeniería disciplinado — de requerimiento a código deployable con criterios objetivos en cada etapa.

01El problema

¿Por qué no alcanza con "vibe coding"?

Sin un proceso definido, la IA introduce variabilidad acumulada en cada paso — de requerimientos vagos a código impredecible sin forma objetiva de medir si el resultado está bien.

⚡

Inconsistencia de output

El mismo requerimiento produce resultados muy diferentes. Sin estructura, no hay garantía de calidad ni coherencia entre sesiones.

🧠

Sin contexto persistente

Cada sesión empieza de cero. El asistente no conoce el stack, las convenciones ni las decisiones previas del proyecto.

📏

Imposible de medir

Sin criterios de aceptación definidos antes de implementar, no hay forma objetiva de evaluar si el output es correcto.

📈

No escala en equipos

Sin estandarización, cada developer usa la IA diferente. Lo que funciona para uno en una tarde no funciona en múltiples equipos.

04Sub-agentes

Los 7 agentes del framework

Cada agente tiene una responsabilidad bien definida, tools restringidos y contexto aislado. Separation of concerns aplicado a la IA.

Coordinador

infra-orchestrator

Punto de entrada. Recibe el requerimiento y delega a los agentes especializados gestionando los gates.

Business Analyst

infra-spec

Traduce requerimientos en specs con criterios de aceptación. Tools: Read, Write, Glob, Grep.

Arquitecto

infra-plan

Diseña el plan de implementación con tareas, dependencias y decisiones técnicas. Tools: + Bash.

Developer

infra-code

Implementa las tareas del backlog siguiendo el plan. Tools: Read, Edit, Write, Bash, Glob, Grep.

Debugger

infra-debug

Diagnostica y resuelve problemas con ciclo hipótesis → validación → fix. Se activa automáticamente.

QA Specialist

infra-validate

Valida la implementación contra los criterios de aceptación. Produce reporte PASS/FAIL por criterio.

Analista

infra-context

Genera y mantiene el Memory Bank. Analiza código y documenta el contexto técnico del proyecto.

¿Por qué sub-agentes dedicados?

Un solo agente ejecutando todos los roles contamina el contexto — el razonamiento de análisis de negocio se mezcla con el de generación de código. La solución es un agente por responsabilidad, ejecutado como sub-proceso aislado con su propia ventana de contexto.

AISLAMIENTO

Cada agente corre con su propio historial. El orquestador recibe solo el output final, sin el ruido del razonamiento intermedio.

LEAST PRIVILEGE

Un infra-spec no puede ejecutar código aunque el prompt se lo pidiera. La restricción es estructural, no de comportamiento del modelo.

PARALELISMO

Sub-agentes independientes pueden correr en background de forma concurrente, acelerando los tiempos de desarrollo.

06Conceptos clave

Los tres pilares del framework

Memory Bank, Skills y Presets trabajan juntos para que el agente sea más efectivo y alineado a la realidad del proyecto.

🧠

Memory Bank

Contexto del proyecto

Dos archivos Markdown en .infracode/memory-bank/ que dan contexto persistente al agente entre sesiones.

project.md
Plataforma, negocio, constraints críticas
technical.md
Stack, arquitectura, convenciones, decisiones
⚙

Skills

Capacidades del agente

Habilidades explícitas y reutilizables que definen propósito, inputs y outputs esperados para cada capacidad.

Embedded
memory-bank, find-skills, skills-guide
De terceros
infracode skills add
📦

Presets

Configuración de plataforma

Paquetes desde GitHub con reglas, workflows y configuraciones específicas para una plataforma tecnológica.

Plataformas
Adobe Commerce · VTEX · Shopify
Instalación
infracode preset add github.com/…

08Workflows

Commands disponibles en el asistente

Una vez inicializado el proyecto, estos workflows están disponibles dentro de tu herramienta de IA.

/infra:spec

Genera spec funcional con criterios de aceptación medibles a partir de un requerimiento en lenguaje natural.

/infra:plan

Genera plan técnico con tareas atómicas, dependencias y decisiones de arquitectura con trade-offs documentados.

/infra:code

Ejecuta las tareas del backlog, escribe tests y corre build y lint después de cada tarea completada.

/infra:validate

QA contra criterios de la spec. Produce reporte PASS/FAIL por cada criterio de aceptación definido.

/infra:debug

Diagnostica bugs con ciclo hipótesis → validación → fix. Se activa automáticamente al fallar tests.

/infra:memory-bank-init

Analiza el código y genera project.md y technical.md. Vos aprobás el contenido antes de que se guarden.

09Primeros pasos

Setup en 4 pasos

De cero a tu primer feature con el flujo SDD en minutos.

1

Configurar el Gateway

infracode gateway --claude

Centraliza acceso a modelos. Seleccionás modelo principal y modelo para sub-agentes. Solo una vez por máquina.

2

Inicializar el proyecto

cd mi-proyecto && infracode init

Wizard interactivo: seleccionás herramientas de IA, plataforma y si querés Memory Bank. Instala los 7 agentes especializados.

3

Iniciar el orquestador

claude --agent infra-orchestrator

El orquestador es el coordinador. Recibe tu requerimiento en lenguaje natural y delega a los agentes especializados.

4

Inicializar el Memory Bank

/infra:memory-bank-init

El agente analiza el código y propone el contexto del proyecto. Vos aprobás. Genera project.md y technical.md en el repositorio.
